# 📘 HispanoComply
# 复评机制 & 版本号规范（内部标准文档）

## 适用范围

- 前端文案（Landing / Result / PDF / Footer）
- 后端评估逻辑输出
- PDF 报告生成
- 后续功能扩展（不破坏用户心智）

---

## 一、产品原则（必须遵守）

### 1️⃣ 复评不是营销动作

- 复评 ≠ 促销
- 复评 ≠ 订阅诱导
- 复评 ≠ 推送提醒
- 复评是一个专业判断行为，仅在经营状态变化时出现。

### 2️⃣ 不存用户行为历史

- 不记录用户第几次评估
- 不记录用户是否复评
- 不对用户行为做时间追踪
- 所有"阶段感"只存在于结果本身，而不是用户身上。

### 3️⃣ 不制造责任暗示

- 不使用"实时 / 最新 / 持续监控"等词
- 不暗示"系统会自动更新判断"
- 不暗示"平台负责合规结果"

---

## 二、复评（Reassessment）语言统一规范

### ✅ 官方定义（唯一版本）

**复评（Reassessment）**
指在经营状态发生变化后，
重新评估当前的合规风险暴露情况。

### ✅ 标准触发条件（唯一清单）

如果以下任一情况发生，建议进行一次复评：

1. 收入连续 1–2 个月出现明显变化
2. 新增员工或长期外包
3. 新增 POS / 平台收款（如 Glovo、Uber Eats、Amazon 等）
4. 经营方式、业务结构或行业检查重点发生变化

⚠️ **任何页面、文案、PDF、提示只能使用这 4 条，不得自行增删或改写含义。**

### ✅ 标准节奏语言（30 天）

- **建议复评时间：30 天**
- 解释补充（可选）：
  - 该周期通常足以完成基础整改，并观察风险是否下降。

### ✅ 心理安全补充语（强制推荐）

复评并不代表你之前的判断是错误的，
而是用于确认当前状态是否发生了变化。

### ✅ 行动清单结尾固定句

完成上述步骤后，
建议进行一次复评，确认风险是否已明显下降。

### ✅ 通用入口标准句（首页 / Footer）

你可以在经营状态发生变化时，
随时进行一次新的复评。

### ✅ PDF 固定用语（封面 & 结尾）

📅 本报告基于生成当时的经营信息与输入数据。

如果你的收入、员工数量或经营方式发生变化，
建议进行一次新的复评。

---

## 三、版本号体系（核心机制）

HispanoComply 使用 **双版本号体系**，用于区分：

1. **这一次评估是什么时候做的**
2. **这一次评估使用的是哪一套规则**

⚠️ **这两个版本号必须区分清楚，不能混用。**

---

## 四、评估版本号（Assessment Version）

### 🎯 用途

- 标识**"这一次判断"**
- 给用户一个时间锚
- 不涉及规则变化

### 📌 规则

评估版本：`vYYYY.MM.DD`

**示例：**
- `v2026.01.03`
- `v2026.02.18`

### 📍 显示位置（统一）

- 结果页（风险分数下方，小字）
- PDF 封面
- PDF 页脚（可选）

### 📄 标准文案

```
评估版本：v2026.01.03
基于该日期的输入与规则生成
```

### 📌 重要说明

- 评估版本 ≠ 系统版本
- 评估版本 ≠ 用户等级
- 评估版本只与时间点相关

---

## 五、规则版本号（Rule Version）【进阶】

### 🎯 用途

标识风险引擎规则集的代际变化

主要用于：
- 内部排查
- PDF 专家版
- 将来法律/规则更新说明

### 📌 规则命名

规则版本：`R{主版本}.{次版本}`

**示例：**
- `R1.0`（初始稳定版）
- `R1.1`（新增行业或信号）
- `R2.0`（重大评分逻辑调整）

### 📌 什么时候升级规则版本

| 情况 | 规则版本变化 |
|------|------------|
| 新增行业 | R1.0 → R1.1 |
| 新增 signals | R1.1 → R1.2 |
| 调整分数权重 | R1.x → R2.0 |
| 文案微调 | ❌ 不升级 |

### 📍 显示策略（非常重要）

- **普通用户页面**：不显示规则版本
- **PDF 专家版**：页脚可显示（小字）

标准格式：

```
规则版本：R1.2
基于当前公开法规与行业经验配置
```

📌 **普通用户不需要理解规则版本，但你为未来留下了可追溯性。**

---

## 六、评估版本 vs 规则版本（对外解释原则）

如果未来用户问起：

**"为什么两次结果不一样？"**

你永远只说这一套逻辑：

> 不同时间的评估，
> 反映的是不同时间点的经营状态与风险判断。
>
> 这属于正常现象，并不代表之前的判断错误。

⚠️
- 不主动提规则变化
- 不主动解释算法
- 避免责任升级

---

## 七、禁止事项（红线）

- ❌ 不在任何地方说"系统会自动更新判断"
- ❌ 不用"实时风险""持续监控"等词
- ❌ 不将规则版本与用户行为绑定
- ❌ 不将评估版本解释为"有效期"
